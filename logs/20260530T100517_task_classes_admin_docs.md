# Task classes, admin editing, and documentation

## What changed

- Updated prompt import so task classes are extracted from markdown `# Classes` and `# Possible classes` sections instead of defaulting every task to yes/no.
- Added prompt-based multi-choice detection for tasks that allow a second label.
- Added task schema validation and annotation validation for unknown class IDs and choice limits.
- Added an admin task-list/update API and expanded the Admin page so superusers can edit task names, prompts, enabled state, multi-choice settings, maximum choices, and class choices.
- Updated the Classification page to render radios for single-choice tasks and checkboxes for multi-choice tasks with frontend validation.
- Added markdown documentation under `docs/` with Mermaid diagrams for architecture and flows.

## Why

The original importer made every task a yes/no task, which did not match the semantic class definitions already present in the prompt markdown files. Administrators also needed an in-app way to correct task metadata without preparing JSON manually.

## Decisions

- Markdown class IDs remain the stable stored class IDs.
- English labels are generated by humanizing class IDs; Czech labels are imported from optional slash labels when present.
- Multi-choice import defaults to a maximum of two classes because current prompts describe optional secondary labels.
