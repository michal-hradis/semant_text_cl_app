## Clarifications & Detailed Design (2026-05-08)

### User workflow
1. User registers (open registration) and logs in.
2. On the task-selection page the user picks **up to 6 classification tasks** from those enabled by admins.
3. The app presents texts one by one. For each text the user sees all selected tasks simultaneously as checkbox groups.
4. Each task is either **single-choice** or **multi-choice** (determined by the task definition).
5. After completing a text the response is uploaded and the next text loads.
6. A counter shows how many texts the user has classified (total and per task).
7. A per-task leaderboard shows top annotators.

### Task definitions
- Tasks, their instructions and valid classes are **stored in the database**.
- The initial import is done by parsing the existing `.md` files in `./prompts/` and producing a canonical JSON representation per task.
- An admin UI panel lets admins **upload a task definition JSON file** (replacing or adding tasks).
- Each task definition contains: `id`, `name`, `description` (Markdown), `multi_choice: bool`, `classes: [{id, label, label_cs}]`, `enabled: bool`.

### Text / data model
- On JSONL upload the **full JSON blob** is stored verbatim in the database. The app extracts `id` and `text` for display; all other fields are available for later export and comparison with AI labels.
- A text may be uploaded multiple times; duplicate `id` is treated as an upsert (update metadata, keep existing annotations).

### Classification data model
- One **annotation** record per `(user_id, text_id, task_id)`.
- Stores: selected class(es), start/end timestamp.
- The app ensures a user does not see the same `(text, task)` pair twice.
- The app selects texts that have the fewest annotations for any of the user's selected tasks (to balance coverage).

### Admin capabilities
- Enable / disable / delete a task.
- Upload texts (JSONL).
- Upload task definition (JSON).
- View annotation statistics.

### Backend refactoring
- Rename Python package `title_annotator` → `text_classifier`.
- Rename DB models: `RatingRequest` → `Text`, `RatingResponse` → `Annotation`.
- Rename API prefix `/api/rating` → `/api`.
- New DB tables: `tasks` (task definitions), `texts` (text items + raw JSON), `annotations`.
- Keep `fastapi-users` for auth; keep JWT bearer strategy.
- Use **SQLite** for development; production can use PostgreSQL via env var `DATABASE_URL`.

### Frontend refactoring
- Rename Quasar app (`package.json` name/productName) to `text-classifier`.
- New pages: `TaskSelectionPage.vue`, `ClassificationPage.vue`, `LeaderboardPage.vue`, `AdminPage.vue`.
- Remove old `RatingPage.vue` and related title-rating components/types once replaced.

### API endpoints (new)
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/tasks` | List all enabled tasks |
| GET | `/api/tasks/{task_id}` | Get task definition (instructions + classes) |
| POST | `/api/admin/tasks` | Upload task definition JSON (admin) |
| PATCH | `/api/admin/tasks/{task_id}` | Enable/disable/delete task (admin) |
| POST | `/api/admin/texts` | Upload texts JSONL (admin) |
| GET | `/api/texts/next` | Get next text for given task_ids |
| POST | `/api/annotations` | Submit annotations for a text |
| GET | `/api/stats/me` | Get current user's annotation counts |
| GET | `/api/stats/leaderboard/{task_id}` | Get leaderboard for a task |

### Tech constraints
- Backend: Python ≥ 3.11, FastAPI, SQLAlchemy async, aiosqlite, pydantic v2, fastapi-users.
- Frontend: Quasar v2, Vue 3 Composition API, TypeScript, Pinia, Axios.
- Tests: `pytest` + `pytest-asyncio`; keep tests in `backend/tests/`.
- Change log: every non-trivial change is summarised in `logs/{iso_datetime}_{task}.md`.
